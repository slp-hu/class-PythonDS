開発側 解答例

F1. DataFrameの [] に渡せるものを整理して説明する

(1) 各コードの型と内容・分類

type(df["A市"])           # → pandas.core.series.Series
type(df[["A市", "B市"]])  # → pandas.core.frame.DataFrame
type(df[df["B市"] > 64])  # → pandas.core.frame.DataFrame
コード 返る型 分類
df["A市"] Series ①列ラベル（文字列）を渡した場合
df[["A市", "B市"]] DataFrame ②列ラベルのリストを渡した場合
df[df["B市"] > 64] DataFrame ③真偽値のSeriesを渡した場合

①と②の違いに注意：文字列を1つ渡すと Series が返るが、 1要素のリストを渡した場合（df[["A市"]]）でも DataFrame が返る。

(2) type(df["B市"] > 64) の確認と仕組みの説明

type(df["B市"] > 64)
# → <class 'pandas.core.series.Series'>

仕組みの説明：

df["B市"] > 64 はまず Series 同士の比較演算として評価される。 pandas は Series の各要素を順番に 64 と比べ、大きければ True、 そうでなければ False を格納した、元と同じ行数（17行）の bool 型 Series を返す。

0     False
1     False
...
12     True   ← B市の値が70なので True
...
16     True
Name: B市, dtype: bool

この bool Series を df[...] に渡すと、pandas は「Series のインデックスと DataFrame のインデックスを照合し、True に対応する行だけを取り出す」という 処理を内部で行う。行の番号（インデックス）をキーにして True/False を参照し、 True の行だけで構成された新しい DataFrame を返す、というのが仕組みである。

補足（query との対比）： 同じ結果は df.query("B市 > 64") でも得られる。 query() は文字列で条件を書けるため、変数名にスペースが含まれない場合は 読みやすいが、列名に日本語や記号が含まれる場合は df.query("`列名` > 64") のようにバッククォートが必要になるなど注意点もある。

F2. bins のデフォルトを使ってはいけない理由と、適切な決め方

(1) デフォルト bins=10 の問題点

売上高データの範囲は 0.33〜104.92 万円（幅 約104万円）である。 bins=10 の場合、ビン幅は 104 ÷ 10 = 約10.4 万円になる。

問題は、この10という値に統計的な根拠がない点にある。 データの個数・ばらつき・範囲を一切考慮していないため、 データが少ない場合は粗すぎて分布の形が見えず、 データが多い場合は細かすぎて棒が乱立し形が読みにくくなる。 「とりあえず10」で描いたヒストグラムは、分布の形を正確に反映しているとは言えない。

(2) 3つの公式による計算（売上高データ、n=100）

import numpy as np

n = len(df)            # 100
std = df["売上高"].std()  # 20.54
q1  = df["売上高"].quantile(0.25)  # 41.36
q3  = df["売上高"].quantile(0.75)  # 71.02
iqr = q3 - q1          # 29.67

# スターゲスの公式（ビン数）
sturges = int(np.ceil(np.log2(n + 1)))
# np.ceil(np.log2(101)) = np.ceil(6.658) = 7
print(f"スターゲス: {sturges} ビン")   # → 7

# スコットの公式（ビン幅）
scott_w = 3.5 * std / n**(1/3)
# 3.5 * 20.54 / 4.642 = 15.49 万円
print(f"スコット: ビン幅 {scott_w:.1f} 万円")  # → 15.5 万円

# フリーマン＝ダイアコニスの公式（ビン幅）
fd_w = 2 * iqr / n**(1/3)
# 2 * 29.67 / 4.642 = 12.78 万円
print(f"FD: ビン幅 {fd_w:.1f} 万円")   # → 12.8 万円
公式 結果
スターゲス 7 ビン
スコット ビン幅 15.5 万円（範囲104万円 ÷ 15.5 ≈ 7 ビン）
フリーマン＝ダイアコニス ビン幅 12.8 万円（範囲104万円 ÷ 12.8 ≈ 8〜9 ビン）

(3) bins の決定とコード

3つの公式の結果は 7〜9 ビンで概ね一致している。 FD公式はIQR（四分位範囲）を使うため外れ値の影響を受けにくく、 本データのように外れ値が存在しうる場合に信頼性が高い。 ここでは FD公式に基づきビン幅を約13万円とし、 データ範囲に合わせた区切りリストを設定する。

# 0万円から110万円を13万円刻みで区切る
bins_sales = list(range(0, 115, 13))
# → [0, 13, 26, 39, 52, 65, 78, 91, 104]

df["売上高"].plot(
    kind="hist",
    bins=bins_sales,
    title="おでんの売上高分布",
    xlabel="売上高（万円）",
    grid=True,
)

なぜ整数刻みに丸めるか： 12.78 万円をそのままビン幅にすると軸ラベルが読みにくくなる。 視認性を考えて13万円に丸め、0万円から始まるリストにすることで 軸の目盛りが自然な値になり、読み手が解釈しやすくなる。

3公式の結果が異なる場合の優先順位： 外れ値が少なくデータが正規分布に近い場合はスコット公式が安定している。 外れ値が疑われる場合はFD公式（IQRベース）が頑健。 スターゲスはあくまでデータ数だけから求めるシンプルな目安であり、 分布の形を考慮しない点で精度は落ちる。迷ったらFD公式を基本とすると良い。

F3. dtype が object のまま分析を始めると何が起きるか

(1) TypeError の原因（補足スクリプトの例）

birth = pd.read_csv("data/chapter02.csv", encoding="Shift-jis")
birth["世帯年間収入"] > 6000  # → TypeError

birth.dtypes を確認すると「世帯年間収入」列が object（文字列型）になっている。 birth["世帯年間収入"][0] を確認すると '4,488'（カンマ付きの文字列）であることがわかる。

CSVファイル内でカンマが桁区切りとして使われていたため、 pandas が数値として認識できず文字列のまま読み込んだ。 文字列（str）と整数（int）の間で > 比較はできないので TypeError が発生する。

解決策はCSV読み込み時に thousands="," を指定することである。

birth = pd.read_csv(
    "data/chapter02.csv",
    thousands=",",      # カンマを桁区切りとして処理
    encoding="Shift-jis"
)
birth.dtypes
# 世帯年間収入    int64  ← 正しく整数型になる

(2) chapter03_exercise.xlsx の dtype 確認

df = pd.read_excel("chapter03_exercise.xlsx")
print(df.dtypes)

出力：

気温     float64
売上高    float64
dtype: object

気温・売上高ともに float64（小数を含む数値型）として正しく読み込まれているため、 そのまま比較演算や統計計算を行うことができる。

(3) もし object になっていた場合の原因と対処法

想定される原因： - セル内にカンマや単位（「円」「万」など）が含まれている - 一部のセルに文字列が混入している（空白セルや「N/A」など） - Excelファイル上でセルが文字列書式に設定されている

対処法（コード例）：

# 方法1：読み込み後に pd.to_numeric() で変換
df["売上高"] = pd.to_numeric(df["売上高"], errors="coerce")
# errors="coerce"：変換できない値は NaN にする

# 方法2：読み込み時に dtype を指定
df = pd.read_excel(
    "chapter03_exercise.xlsx",
    dtype={"気温": float, "売上高": float}
)

# 変換後に NaN が発生していないか確認する
print(df.isnull().sum())

実務上の教訓： データを読み込んだら、分析を始める前に必ず df.dtypes と df.head() を確認する習慣をつける。 「計算できた数値」が実は文字列の並びを操作したものだった、 というミスは静かに起きるため、実行時エラーよりも発見が難しい。

分析・企画側 解答例

G1. 平均・中央値・標準偏差は何を「代表」しているのか

(1) ばらつきの比較と経営上の意味

相対的なばらつきを比べるために変動係数（CV = 標準偏差 ÷ 平均）を使う。

標準偏差 平均 変動係数
気温 6.53℃ 20.25℃ 32.3%
売上高 20.54万円 56.03万円 36.7%

売上高の方がばらつきが大きい。

経営上の意味：売上が日によって大きく変動することを示している。 「いつも同じくらい売れる」のではなく、「よく売れる日と売れない日の差が大きい」。 これは在庫管理や仕入れ計画を難しくする要因であり、 売上を安定させるための施策（気温連動の仕入れ調整など）が重要になることを示唆する。

(2) 代表値の選択

中央値（58.80万円）を使う方が適切。

理由：売上高データでは平均（56.03万円）より中央値（58.80万円）の方が大きい。 これはデータが左に裾を引いている、つまり売上が極端に低い日が 少数あって平均を引き下げていることを意味する （最小値が0.33万円と非常に低いことからも確認できる）。

平均はこうした外れ値の影響を受けやすいため、 「典型的な日の売上」を表す指標としては中央値の方が実態に近い。

平均を使うと「普通の日の売上は約56万円」と伝えることになるが、 実際には半数以上の日が58万円以上売れており、やや過小評価になる。

(3) 1つの数値では伝わらないこと

  • ばらつきの大きさ：平均や中央値は「中心」を示すだけで、日々の変動幅はわからない。標準偏差（20.54万円）もあわせて示さないと、「安定して56万円売れる」と誤解される恐れがある。
  • 分布の形（偏り・外れ値）：最小値が0.33万円と極端に低い日が存在することは、平均・中央値だけでは読み取れない。ヒストグラムや最小・最大値を示して初めてわかる。

G2. ヒストグラムは「誰のため」に描くのか

(1) 自己理解のための読み取り（3点）

  1. 分布のピーク帯：売上高は60〜80万円付近に最も多く集中しており、この帯が「よくある売上日」である。
  2. 低売上日の存在：0〜20万円台にも数日分のデータがあり、極端に売れない日がごく少数ある。最小値は0.33万円と、ほぼ売上ゼロの日も存在する。
  3. 分布の偏り（左裾）：高売上側（80〜100万円超）よりも低売上側（0〜30万円台）に外れた値が多く、分布は左に引きずられる形をしている。これが平均が中央値より低い原因である。

(2) 他者（業者）への説明での見せ方の工夫

自己理解のためのヒストグラムのままでは、見せる相手がデータ分析に慣れていない場合に伝わりにくい。以下の点を工夫する。

df["売上高"].plot(
    kind="hist",
    bins=list(range(0, 115, 13)),
    title="おでん1日あたり売上高の分布（過去100日）",  # 期間を明記
    xlabel="売上高（万円）",
    ylabel="日数（日）",                              # 縦軸の意味を明記
    grid=True,
    color="steelblue",
    edgecolor="white",
)
plt.axvline(df["売上高"].median(), color="red",
            linestyle="--", label=f"中央値 {df['売上高'].median():.0f}万円")
plt.legend()
  • タイトルに「過去100日」など調査期間を入れ、いつのデータかを明示する
  • 縦軸のラベルを「日数（日）」と入れ、棒の高さが何を表しているかを明示する
  • 中央値の補助線を引き、「標準的な日の売上」がひと目でわかるようにする

(3) ヒストグラム1枚に加えてほしいグラフや情報

  • 気温と売上高の散布図：気温と売上の関係（相関）が視覚的に確認でき、「低気温の日ほど売れる」という仮説の裏付けになる。
  • 日付順の折れ線グラフ（時系列）：季節的な傾向（夏は売れない、冬は売れるなど）がヒストグラムでは見えない。時系列で並べることで傾向が把握できる。

G3. 「特徴を踏まえて計画を立てる」ために何が必要か

(1) 言えること・言えないこと

確実に言えること（2点）： - 過去100日間のおでん売上高は、半数の日が58.8万円以上、半数が58.8万円未満だった（中央値による事実）。 - 売上高には大きなばらつきがあり（標準偏差20.5万円）、最低0.33万円〜最高104.9万円と約100万円の幅で変動した。

言えそうで言えないこと（2点）： - 「これからも同じ傾向が続く」とは言えない。100日分のデータがどの時期のものか不明であり、季節変動や年次変化を考慮していない。 - 「気温が低いから売上が高い」という因果関係は言えない。相関があっても因果は別問題であり、他の要因（曜日・イベント・競合など）が影響している可能性がある。

(2) 仕入れ計画への活用と、使えなかった情報

活用できる判断プロセス：

  1. 中央値58.8万円を「ベースの仕入れ量」の基準にする（平均ではなく中央値を使う理由：外れ値に引きずられない）。
  2. 標準偏差20.5万円を踏まえ、売上が「中央値 ± 20万円（38〜79万円）」の範囲に収まる確率が約7割と想定し、その幅をカバーできる在庫を準備する。
  3. 売上高0〜20万円台の日が数日存在することを念頭に、廃棄リスクが高い仕入れは避ける。

使えなかった情報（2点）： - 気温予報との対応関係：気温と売上が実際にどう連動しているかが数値でわかれば、翌日の気温予報から仕入れ量を調整できる。今回のデータ分析だけではその対応関係を定量化できていない。 - 曜日・日程の情報：週末や祝日・イベント日に売上が変動するかどうかが不明。仕入れ計画を立てるには日付情報が必要だが、今回のデータにはない。

(3) 店長への口頭報告（1分）

「過去100日分のおでんの売上データを分析しました。 典型的な日の売上は約59万円ですが、日によって大きくばらつきがあり、 最低0.33万円から最高104万円まで約100万円の差がありました。

売上の高い日と低い日の差が非常に大きいため、 毎日同じ量を仕入れるのではなく、気温予報などに合わせて仕入れ量を 調整できると、廃棄ロスを減らせる可能性があります。

ただし、今回は気温と売上の関係や、曜日・季節ごとの傾向まではわかっていないため、 追加のデータ分析が必要です。」
